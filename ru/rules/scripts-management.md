# Scripts Management — Управление скриптами

---

## Scripts Management [MANDATORY]

### Скриптовые расположения

Скрипты могут находиться в нескольких папках. Определить перечень скриптовых расположений при деплое.

**Единые файлы управления** (хранятся в основном `scripts/`):
- `REGISTRY.md` — **единый реестр** всех скриптов из всех расположений
- `TRIVIAL_TASKS.md` — общий список типов тривиальных задач
- `SPEC_TEMPLATE.md` — базовый шаблон spec-файла
- `logs/` — единая папка логов для локальных скриптов
- `archive/` — единая папка для устаревших скриптов

**Все скрипты, создаваемые AI-ассистентом [MANDATORY]**: независимо от расположения, **обязаны** проходить полный процесс Scripts Management: spec-файл, шапка, реестр.

### 1. Структура папок

```
scripts/
├── REGISTRY.md
├── TRIVIAL_TASKS.md
├── SPEC_TEMPLATE.md
├── logs/
│   └── {имя_скрипта}_{YYYYMMDD}_{HHMM}.log
├── archive/
├── {скрипт}.py|sh
├── {скрипт}_spec.md
└── ...
```

**Принципы:**
- Скрипт и его spec-файл лежат **рядом**, в одной папке
- `scripts/logs/` — единая папка логов для всех локальных скриптов
- `scripts/archive/` — единая папка для устаревших скриптов (перемещать парой: скрипт + spec)
- **При перегруппировке или перемещении скриптов [MANDATORY]**: обновить пути во всех местах:
  1. Обновить столбец «Путь» в `scripts/REGISTRY.md`
  2. Обновить поле `SPEC:` в шапке скрипта (если путь к spec-файлу изменился)
  3. Обновить поле «Расположение скрипта» в spec-файле (если есть)
  4. Обновить любые перекрёстные ссылки в других документах

### 2. Spec-файл (требования к скрипту)

**Именование**: `{имя_скрипта}_spec.md`

Пример: `transcribe_video.py` → `transcribe_video_spec.md`

**Содержание**: по шаблону из папки скрипта (`SPEC_TEMPLATE_VPS.md`, `SPEC_TEMPLATE_*.md`), если есть; иначе по базовому `scripts/SPEC_TEMPLATE.md`

**Обязательные элементы spec-файла:**
- Назначение (одно предложение)
- Входные данные (аргументы, файлы, переменные окружения)
- Выходные данные (что создаёт, куда пишет)
- Зависимости (пакеты, утилиты, API)
- Поведение (логика, обработка ошибок, ограничения)
- Логирование (куда пишет лог, что логирует)
- **Уровень контроля**: `АППРУВ ТРЕБУЕТСЯ` или `ТРИВИАЛЬНАЯ ЗАДАЧА (без аппрува)`
- **Режим запуска** [CANONICAL]:
  - Тип: `manual` / `auto:systemd` / `auto:cron` / `called`
  - Для `auto:systemd` — имя сервиса, путь к unit-файлу
  - Для `auto:cron` — расписание cron
  - Для `called` — какой скрипт вызывает и при каких условиях
- История изменений требований (таблица)
- Три даты в конце:
  - **Дата создания требований** — не меняется
  - **Дата актуализации требований**
  - **Дата приведения скрипта в соответствие**

### 3. Шапка скрипта (ссылка на spec)

**Python:**
```python
#!/usr/bin/env python3
"""
Краткое описание назначения скрипта.
SPEC: script_name_spec.md
COMPLIANT: YYYY-MM-DD
LAUNCH: manual
"""
```

**Bash:**
```bash
#!/bin/bash
# Краткое описание назначения скрипта.
# SPEC: script_name_spec.md
# COMPLIANT: YYYY-MM-DD
# LAUNCH: manual
```

`LAUNCH:` допустимые значения:
- `manual`
- `auto:systemd(имя.service)`
- `auto:cron(расписание)`
- `called:имя_скрипта`
- Комбинации через запятую

### 4. Процесс создания скрипта

1. **Определить уровень контроля** — сверить со списком `scripts/TRIVIAL_TASKS.md`
2. **Сформулировать требования** → написать spec-файл по шаблону
3. **Если аппрув требуется** → показать пользователю и ждать подтверждения
4. **Создать скрипт** с шапкой `SPEC:` + `COMPLIANT:` + `LAUNCH:`
5. **Обновить `scripts/REGISTRY.md`**
6. **Проставить все даты** — в spec-файле и в шапке скрипта
7. **Если auto:systemd или auto:cron** — создать/обновить unit-файл или cron-запись

### 5. Процесс изменения скрипта

1. **Прочитать текущий spec-файл**
2. **Обновить требования** в spec-файле
3. **Обновить «Дата актуализации требований»**
4. **Добавить запись** в таблицу «История изменений требований»
5. **Если аппрув требуется** → показать пользователю изменённые требования и ждать подтверждения
6. **Внести изменения в скрипт**
7. **Обновить `COMPLIANT:`** в шапке и дату в spec-файле
8. **Обновить `LAUNCH:`** если изменился режим запуска
9. **Обновить `scripts/REGISTRY.md`**

### 6. Реестр скриптов (REGISTRY.md)

Таблица со столбцами:
- **Скрипт** — имя файла
- **Путь** — полный относительный путь
- **Назначение** — одно предложение
- **Язык** — Python / Bash
- **Запуск** — режим запуска
- **Статус** — активен / архив / в разработке
- **Область** — `системный` / `проектный`
- **Дата создания**

Обновлять при: создании, изменении назначения, перемещении, архивации, изменении режима запуска скрипта.

### 7. Логирование выполнения скриптов

**Папка**: `scripts/logs/`
**Именование**: `{имя_скрипта}_{YYYYMMDD}_{HHMM}.log`

**Что логировать:** время старта, входные параметры, ключевые этапы, результат, время завершения.

**Python — шаблон:**
```python
import logging
from datetime import datetime
from pathlib import Path

script_name = Path(__file__).stem
log_dir = Path(__file__).parent / "logs"
log_dir.mkdir(exist_ok=True)
log_file = log_dir / f"{script_name}_{datetime.now().strftime('%Y%m%d_%H%M')}.log"

logging.basicConfig(
    level=logging.INFO,
    format="%(asctime)s [%(levelname)s] %(message)s",
    handlers=[
        logging.FileHandler(log_file, encoding="utf-8"),
        logging.StreamHandler()
    ]
)
```

**Bash — шаблон:**
```bash
SCRIPT_NAME=$(basename "$0" .sh)
LOG_DIR="$(dirname "$0")/logs"
mkdir -p "$LOG_DIR"
LOG_FILE="$LOG_DIR/${SCRIPT_NAME}_$(date +%Y%m%d_%H%M).log"
log() { echo "$(date '+%Y-%m-%d %H:%M:%S') [$1] $2" | tee -a "$LOG_FILE"; }
log "INFO" "Старт: $0 $*"
```

### 8. Список тривиальных задач

**Файл**: `scripts/TRIVIAL_TASKS.md`

Содержит перечень типов/категорий задач, для которых spec-файл создаётся **без подтверждения пользователя**. Пополняется только после явного согласия пользователя (см. п.4, шаг 1).

### 9. Самопроверка после изменения скрипта [CRITICAL]

| # | Пункт | Выполнено? |
|---|-------|-----------|
| 1 | Текущий spec-файл прочитан перед началом работы | |
| 2 | Требования в spec-файле обновлены | |
| 3 | «Дата актуализации требований» обновлена | |
| 4 | Запись добавлена в таблицу «История изменений» | |
| 5 | Аппрув получен (если «АППРУВ ТРЕБУЕТСЯ») | |
| 6 | Изменения внесены в скрипт | |
| 7 | `COMPLIANT:` обновлён в шапке + дата в spec-файле | |
| 8 | `LAUNCH:` обновлён (если изменился режим запуска) | |
| 9 | `scripts/REGISTRY.md` обновлён | |

**Правила:**
- Чеклист заполняется **в TODO-листе или в ответе пользователю** — до перехода к следующей задаче
- Если какой-то пункт не выполнен — **остановиться и выполнить**, прежде чем двигаться дальше
- Пункт 9 (REGISTRY.md) обязателен **всегда**, даже если изменения кажутся незначительными — обновлять хотя бы дату актуализации

### 10. Аппрув требований ≠ одобрение плана [CRITICAL]

**Одобрение плана действий пользователем НЕ заменяет аппрув требований (spec-файла).**

- Если скрипт имеет уровень контроля `АППРУВ ТРЕБУЕТСЯ`:
  1. **Сначала** обновить spec-файл с новыми требованиями
  2. **Затем** показать пользователю **конкретные изменения в spec-файле** и явно спросить: «Требования в spec-файле верны? Можно приступать к реализации?»
  3. **Только после явного подтверждения** — приступать к изменению кода скрипта
- Согласие пользователя на план, стратегию, архитектуру или описание шагов — **не считается** аппрувом требований
- Аппрув требований — это подтверждение **конкретного содержимого spec-файла** после его обновления
